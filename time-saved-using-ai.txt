========================================
Started: March 2026
Developer: Daniel, Steadfast Code, Lititz PA
Context: Building in spare time while working full-time and raising a family.
         A few hours a day, evenings and weekends.

SESSION 1 — March 12, 2026
---------------------------
What was built:
  - Full backend scaffolding (Express, Mongoose, MongoDB)
  - RAG pipeline (scraper, chunker, retrieval)
  - Vector embeddings (Voyage AI) with smart rescrape (hash-based diffing)
  - Claude API integration with full system prompt injection
  - Temporal context injection (timestamps, returning visitor awareness)
  - Deduplication of scraped chunks
  - Retry logic with exponential backoff for rate limits
  - Vanilla JS chat widget shell
  - MongoDB Atlas Vector Search integration
  - Naming architecture decisions (Entity vs Business, etc.)

Estimated time for an experienced backend developer who knows all these
technologies: 2-3 weeks at 40hrs/week.

Estimated time for Daniel building solo with AI assistance for questions,
but coding manually: 3-6 months at 40hrs/week.

Estimated time for Daniel building in spare time (few hours/day after work
and family): 6-12 months of evenings and weekends. Possibly longer.

What actually happened: ~1 day.

---

Key insight:
The real value isn't just speed. It's architectural decisions made correctly
on the first pass — vector embeddings over keyword matching, hash-based
diffing for rescrapes, proper separation of concerns in the service layer.
These are the calls that come from experience. Getting them right upfront
instead of refactoring later is where the time savings compound most.

The emotional toll of solo learning is real too. Nights spent stuck on
something small, making no visible progress, deciding whether the sacrifice
is worth it — that's where passion projects die. AI removes that wall.

---

Running Total: ~6-12 months of evenings/weekends saved in session 1 alone.

SESSION 7 — March 21-22, 2026 (started ~midnight, ended ~4 AM)
---------------------------------------------------------------
What was built:
  - Paragraph-aware chunking — complete rewrite of text extraction using
    recursive DOM traversal (extractStructuredText) emitting \n\n around
    block elements; chunker splits on \n+ (handles Puppeteer + cheerio);
    sentence-boundary overlap; CHUNK_TARGET=1200, CHUNK_MAX=1800
  - Paragraph-level hash dedup (seenParaHashes shared across batches) —
    strips nav/footer boilerplate before chunking more reliably than
    chunk-level dedup alone
  - Model routing simplification — Church Mode → Sonnet, standard → always
    Haiku; returns { text, model } for analytics
  - Per-entity RAG threshold slider (0.50–0.95, default 0.75), superadmin-gated
  - Model routing analytics — model/topScore/hadContext stored per message,
    30-day stats endpoint, Settings RAG Tuning card
  - KB chunk viewer — accordion shows pages, click to lazy-load chunks,
    expand individual chunk content; dark mode fix
  - Force rescrape (superadmin-gated) — wipes scraped chunks, re-embeds from
    scratch; $nin guard preserves manual/upload/owner_reply chunks
  - Auto-refresh on scrape — KB and chunk viewer update live during scrape_progress,
    full reload on scrape_complete
  - Owner reply KB visibility — knowledge route includes owner_reply chunks
    alongside manual/upload entries
  - Two-phase RAG tree retrieval — Phase 1: Atlas Vector Search at primary
    threshold; Phase 2: all chunks from matched pages scored in-memory via
    dot product (Voyage embeddings are unit-normalized), included if ≥
    max(0.50, threshold − 0.15); chunkIndex stored on chunks for future
    ±N neighbor narrowing once semantic chunking lands
  - Architecture discussion: hashed out why tree retrieval beats LLM-based
    look-ahead, why semantic chunking and tree retrieval are synergistic,
    and the right shape of embedding-based (not LLM-based) semantic chunking
  - Billing UI, notifications UI, MinistryRequests view, BibleVerse model
  - Branch renamed master → main

Estimated additional time saved: 2-3 months of evenings.
  (The RAG quality work alone — diagnosing the Camp Calvary retrieval miss,
  tracing it to semantic mismatch in embeddings, designing and implementing
  two-phase tree retrieval — is the kind of debugging + architecture work
  that takes experienced engineers days. The chunker rewrite, chunk viewer,
  and force rescrape round out a session that meaningfully advances the
  product's reliability before any real users arrive.)

Running Total: ~17-30 months of evenings/weekends saved.
